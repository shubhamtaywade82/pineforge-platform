# RE10143 — Historical offset beyond buffer limit

Source: https://www.tradingview.com/pine-script-docs/errors/RE10143/

**Type:** Runtime error

**Message:** The requested historical offset (X) is beyond the historical buffer's limit (Y).

## Cause

Pine sizes buffers from historical-bar references (first 244 bars, then restart if needed). **Realtime bars do not resize buffers.** Deeper `[]` on realtime than on history → error.

```pine
// Fails on first realtime tick — history used offset 500, realtime uses 1000
float myVar = close[barstate.ishistory ? 500 : 1000]
```

## Fixes

### 1. max_bars_back() — preferred for one series

```pine
float myVar = close[barstate.ishistory ? 500 : 1000]
max_bars_back(close, 1000)
```

Can appear anywhere in script. Max 5000 bars (10000 for OHLC/time).

### 2. max_bars_back parameter — all series

```pine
indicator("...", max_bars_back = 1000)
```

Use sparingly — sizes **every** buffer; hurts memory/perf.

### 3. Larger offset on early bars

```pine
float myVar = close[barstate.isfirst ? 1000 : barstate.ishistory ? 500 : 1000]
```

Forces buffer creation during historical load (offset on bar 0 returns `na`).

## Drawings and time series

Drawings with `bar_index` coords convert to timestamps via `time[]` internally.

```pine
// Fails realtime — line.new(x1 = bar_index - 500, ...) needs time[500]
if barstate.isrealtime
    line.new(bar_index - 500, close, bar_index, close)

max_bars_back(time, 500)
```

## Codegen rules

- Use the **same max lookback** on historical and realtime bars, or set `max_bars_back`
- Any drawing anchored N bars back on realtime → `max_bars_back(time, N)`
- Prefer consistent offsets over `barstate.ishistory` branching when possible
- See also `docs/pine_v6_writing/limitations.md` for buffer limits (5000/10000)
