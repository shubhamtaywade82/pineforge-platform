# RE10139 — Memory limits exceeded

Source: https://www.tradingview.com/pine-script-docs/errors/RE10139/

**Type:** Runtime error

**Message:** Memory limits exceeded.

## Common causes

1. **`request.*()` returning collection/object IDs** — each bar stores a copy
2. Excessive `request.*()` calls (especially `request.security_lower_tf()`)
3. Oversized `max_bars_back` / unnecessary buffer sizing
4. Drawing redraws and historical table/label updates
5. Strategy with too many orders on long history

## Primary fix: optimize request.*()

### Return last state only

```pine
dataFunction() =>
    var array<float> dataArray = array.new<float>(0)
    float bop = (close - open) / (high - low) * 100
    dataArray.push(bop)
    [bop, barstate.islast ? dataArray : na]

[latestValue, reqData] = request.security(syminfo.tickerid, "1D", dataFunction())
```

Only last bar copies the array into main context.

### Return IDs on some bars

Return object ID when needed (e.g. first bar of month), `na` otherwise.

### Return calculated results (best)

```pine
dataFunction() =>
    var array<float> dataArray = array.new<float>(0)
    dataArray.push(bop)
    [bop, dataArray.slice(...).avg()]  // no array ID returned

[latestValue, avgBOP] = request.security(syminfo.tickerid, "1D", dataFunction())
```

## Other fixes

| Issue | Fix |
|-------|-----|
| Too many requests | Merge tuple requests; use `calc_bars_count` |
| LTF intrabar arrays | Higher timeframe or fewer bars |
| `max_bars_back` too large | Size only needed series via `max_bars_back()` |
| Table updates every bar | `var table` + update on `barstate.islast` only |
| Redraw lines/boxes | Use `line.set_*()` / `box.set_*()` |
| Strategy order flood | Filter start date for order logic |

## Codegen rules

- `request.security()` → return floats/tuples of scalars, not arrays/maps
- Tables: create once with `var`, populate on `barstate.islast`
- Drawings: update setters, don't delete+recreate each bar
- Set `calc_bars_count` in declaration when full history not needed
